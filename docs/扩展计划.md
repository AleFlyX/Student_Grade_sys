# 学生成绩查询系统扩展计划

## 1. 文档目的

本文档基于 [use_cases_spec.md](use_cases_spec.md) 中的 25 个用例，对当前“学生成绩查询系统 MVP”进行扩展规划，目标是把现有的“学生登录 + 成绩查询”能力，逐步演进为覆盖学生、教师、管理员三类角色的完整教务业务系统。

## 2. 当前状态与差距

### 2.1 当前已有能力

当前仓库中的系统主要实现了以下能力：

- 学生登录
- 学生成绩查询
- 按学年、学期筛选成绩
- 成绩总学分与 GPA 统计
- 前后端分离基础结构
- 基于 Docker 的后端部署能力

### 2.2 与用例规约的差距

与 [use_cases_spec.md](use_cases_spec.md) 相比，当前系统仍缺少这些核心模块：

- 学生注册、个人信息维护、退出登录、修改密码
- 教师登录、课程管理、学生名单查看、成绩录入与修改
- 管理员登录、学生/教师/课程管理、教师分配、成绩审核与报表管理
- 成绩申诉、成绩统计分析、成绩单打印等扩展功能
- 选课数据关联、补考/重修标记、成绩等级、教务审核流转

## 3. 扩展目标

扩展后的系统需要满足以下目标：

1. 支持学生、教师、管理员三类角色统一登录与权限控制。
2. 支持完整的成绩数据生命周期，包括录入、修改、审核、发布、查询与统计。
3. 支持课程、选课记录、成绩记录之间的关联查询。
4. 支持补考、重修、成绩状态、成绩等级等大学场景特性。
5. 支持成绩统计、报表导出、成绩单打印和成绩申诉等增强功能。

## 5. 分阶段实施计划

### 5.1 Phase 0：数据模型与权限基础

目标是先把系统底座补齐，否则后续功能很难稳定扩展。

建议完成内容：

- 引入统一用户表与角色字段，区分学生、教师、管理员。
- 补充学生信息表、教师信息表、课程表、开课记录表、选课记录表、成绩表。
- 增加成绩审核状态、成绩状态、补考/重修标记、成绩等级字段。
- 统一 JWT 鉴权、路由权限控制与接口返回格式。
- 明确密码加密方式和账号唯一性规则。

交付结果：

- 数据库结构可支撑多角色业务。
- 后端接口拥有统一认证与鉴权基础。

### 5.2 Phase 1：学生端扩展

目标是把学生端从“只查成绩”扩展为“完整个人成绩门户”。

建议完成内容：

- 学生注册与登录联动。
- 学生成绩列表支持更多筛选条件：学年、学期、课程性质、成绩状态。
- 增加成绩详情页，展示分项成绩、教师评语、补考/重修标记。
- 增加 GPA、总学分、获得学分、学期统计等汇总卡片。
- 增加修改密码、退出登录、个人信息查看入口。

交付结果：

- 学生可完成注册、登录、查分、改密、退出等基础闭环。

### 5.3 Phase 2：教师端扩展

目标是支持教师对课程与成绩的日常管理。

建议完成内容：

- 教师登录后进入课程主页。
- 查询所授课程列表与课表信息。
- 查看某门课程的选课学生名单。
- 支持成绩批量录入、暂存、提交审核。
- 支持成绩修改与修改日志记录。
- 支持课程成绩统计与导出。

交付结果：

- 教师可完成“查课-录分-提审-统计”的完整业务流。

### 5.4 Phase 3：管理员端扩展

目标是完成系统级管理与审核能力。

建议完成内容：

- 管理员登录与后台首页。
- 学生、教师、课程的增删改查。
- 课程与教师分配关系管理。
- 成绩审核、退回、发布。
- 系统选课情况统计与成绩报表管理。
- 支持按学期、按课程、按专业导出报表。

交付结果：

- 管理员可完成账号、课程、成绩和报表的统一管理。

### 5.5 Phase 4：增强与优化

目标是补齐更接近真实学校场景的能力。

建议完成内容：

- 成绩申诉流程。
- 成绩单 PDF 导出与防伪信息。
- 学业预警与成绩分析图表。
- GPA 计算规则配置化。
- 补考、重修、成绩等级的更细粒度展示。

交付结果：

- 系统从“可用”升级为“更贴近教务场景”。

## 6. 模块拆分建议

### 6.1 后端模块

- 认证模块：登录、注册、退出、改密、Token 校验
- 学生模块：个人信息、成绩查询、成绩统计、申诉
- 教师模块：课程列表、学生名单、成绩录入、成绩修改、统计导出
- 管理员模块：用户管理、课程管理、分配管理、审核发布、报表管理
- 公共模块：权限校验、统一响应、分页、日志、文件导出

### 6.2 前端模块

- 登录注册页
- 学生成绩主页
- 教师课程与成绩页
- 管理员后台页
- 通用组件：筛选栏、统计卡片、表格、弹窗表单、导出按钮

### 6.3 数据库模块

- 用户与角色
- 学生信息
- 教师信息
- 课程与开课记录
- 选课记录
- 成绩记录
- 成绩审核记录
- 申诉记录
- 操作日志

## 7. 接口扩展建议

在现有 `/auth/login` 与 `/grades` 基础上，建议逐步补充以下接口：

- `/auth/register`
- `/auth/logout`
- `/auth/change-password`
- `/students/me`
- `/students/me/grades`
- `/teachers/me/courses`
- `/teachers/courses/{courseId}/students`
- `/teachers/courses/{courseId}/grades`
- `/admin/users`
- `/admin/courses`
- `/admin/audits`
- `/reports/grades`
- `/appeals`

## 8. 验收标准

### 8.1 功能验收

- 学生、教师、管理员能够按角色登录并访问各自页面。
- 学生成绩查询支持筛选、统计和明细展示。
- 教师可录入并更新成绩，管理员可审核发布。
- 管理员可维护用户、课程和报表。
- 成绩申诉与成绩导出流程可闭环。

### 8.2 数据验收

- 所有核心数据表之间存在清晰外键或逻辑关联。
- 成绩、审核、申诉、操作日志可追踪。
- GPA 和总学分统计结果与测试数据一致。

### 8.3 非功能验收

- 登录与成绩查询接口响应稳定。
- 角色权限隔离有效，学生不能访问教师和管理员功能。
- 关键数据修改操作具备日志记录。

## 9. 推荐实施顺序

如果希望控制开发风险，建议按以下顺序推进：

1. 先补数据模型和权限系统。
2. 再完善学生端闭环。
3. 然后扩展教师端成绩管理。
4. 接着实现管理员审核与报表。
5. 最后补申诉、成绩单、学业分析等增强能力。

## 10. 结论

当前项目已经具备成绩查询系统的 MVP 骨架，但若要完整覆盖 [use_cases_spec.md](use_cases_spec.md) 的需求，需要按“数据底座 -> 学生端 -> 教师端 -> 管理端 -> 增强功能”的顺序逐步扩展。这样可以避免一次性重构过大，同时保证每个阶段都能独立验收。
